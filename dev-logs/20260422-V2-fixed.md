# 2026-04-22 V2 Audit Fix Notes

## Broker Reconciliation Before Entries

Implemented:

```text
runtime/reconciliation.py filters MT5 positions by System C magic number
runner.broker_snapshot() runs before every live evaluation loop
runner reconciles broker positions against state.open_trades before symbol evaluation
broker-closed state trades are removed from state and saved immediately
broker-orphan live positions are recorded in diagnostics and block new entries
history_deals_get(position=ticket) is used for close reconstruction
close rows are append-only with decision=broker_closed
live open rows keep close_time/r_result blank with decision=live_open
```

Close row contract:

```text
close_time filled from MT5 deal history when available
exit_price filled from the exit deal when available
exit_reason inferred as SL, TP, MANUAL, or UNKNOWN
r_result computed from state/history entry, exit price, direction, and SL distance
manual broker closes are classified as MANUAL when MT5 reports CLIENT/MOBILE/WEB/EXPERT/manual reason
```

## Performance Fix: GPS Cadence

Decision:

```text
GPS should not run every 5-second trading loop
GPS is trade-log based, not indicator-clock based
set GPS loop cadence to 5 minutes because the lowest strategy bar resolution is 5m
force GPS immediately when broker reconciliation records a closed trade
```

Implemented:

```text
config/config.yaml gps.loop_interval_seconds=300
config/config.yaml gps.run_on_trade_close=true
runtime/runner.py writes GPS only when the interval is due or a broker close happened
diagnostics records last_gps_run_time, last_gps_run_reason, and last_gps_skip_reason
```

Deferred:

```text
incremental feature rebuild can wait until real loop timing shows pressure
CSV/fsync durability can wait because correctness and close reconciliation are now the higher-risk boundary
```

## Deferred Performance Items Rationale

### Incremental feature rebuild

Why deferred:

```text
runtime/data_cache.py already does the important low-risk optimization:
  startup warms 500 closed bars per required symbol/timeframe
  running loops probe the latest closed bar first
  if no new closed bar exists, symbol evaluation is skipped
  if a new closed bar exists, only a small overlapping delta is fetched into the cache

The remaining cost is inside runtime/engine_bridge.py / copied engine code:
  features are rebuilt from the cached DataFrame when a symbol has a new entry bar
  this is CPU work, not repeated full MT5 history polling
  Numba is already configured for the heavy supertrend path

At the current portfolio size of 4 symbols and 3-4 timeframes per symbol, the
correctness risk of changing feature statefulness is higher than the expected
performance gain before live timing data exists.
```

What would justify doing it:

```text
symbol_evaluation timing regularly exceeds the 5m entry bar budget
Numba feature rebuild becomes the dominant stage in timing logs
MT5 loop falls behind closed-bar updates during London/New York active periods
status/logs show repeated data freshness lag while CPU is saturated
```

Preferred later shape:

```text
keep BarCache as the owner of closed OHLCV frames
add a per-symbol/timeframe feature cache keyed by latest_closed_bar
only rebuild the trailing window required by ATR/supertrend/session features
keep a full-rebuild fallback for startup, corruption, and cache mismatch
add parity tests: incremental output must match full rebuild on the same bars
```

### CSV fsync durability

Why deferred:

```text
state durability is already the critical boundary:
  state writes are atomic through temp file + os.replace
  live order state is saved immediately after a successful order_send and ticket
  broker reconciliation can recover broker/state drift on the next loop

CSV rows are currently append-only observability artifacts:
  open rows document live_open/paper_open decisions
  close rows document broker_closed outcomes
  GPS reads closed rows with close_time and r_result

Adding fsync to every CSV append improves power-loss durability, but it also adds
I/O cost and more code paths around every event/signal/timing row. That is not
the main live safety risk now; broker close detection and state consistency are.
```

What would justify doing it:

```text
trade log loss is observed after process crash or machine reboot
GPS/reporting must be treated as audit-grade instead of operational guidance
the VPS filesystem or sync layer shows delayed flush behavior
we decide trade CSV is a source of truth equal to state + MT5 history
```

Preferred later shape:

```text
fsync only critical logs first: trade and state_audit
leave high-volume signal/timing/event logs buffered unless debugging requires fsync
write through a small durability helper so schema/header logic stays unchanged
consider periodic batched fsync for non-critical logs
verify crash behavior with a fake writer test before enabling in live runtime
```

## VPS Hotfix: MT5 Rates Shape

Symptom:

```text
VPS native MT5 run failed in runtime/data_cache.py with:
ValueError: MT5 rates are missing time column
```

Difference from legacy:

```text
legacy run_data.py passed the native MT5 rates object directly into pd.DataFrame
V2 adapters converted MT5 rates to list(...) before DataFrame construction
on native Windows MT5 this can lose numpy structured-array field names
the resulting DataFrame has numeric columns instead of time/open/high/low/close
```

Fix:

```text
runtime/adapters.py now returns the native/materialized rates object instead of forcing list(...)
runtime/data_cache.py now accepts MT5 structured arrays through dtype.names
runtime/data_cache.py also recovers standard unnamed MT5 tuple rows using the known MT5 rate columns
tests/test_data_cache.py covers structured-array and tuple-row inputs
```

## VPS Console Heartbeat

Reason:

```text
V2 originally logged startup/loop status to CSV only
legacy V1 printed startup status and ALIVE lines to terminal
on VPS, silent terminal makes it hard to know whether the bot is alive or stuck
```

Implemented:

```text

runtime/runner.py prints Main loop started
runtime/runner.py prints ALIVE heartbeat every runtime.heartbeat_minutes
ALIVE includes loop id, mode, equity, balance, open trades, broker positions, evaluated/skipped/candidate counts, duration, invariant, GPS, and blocked flag
```

## Market Data Stale Console Warning

Implemented:

```text
config/config.yaml runtime.market_data_stale_minutes=180
config/config.yaml runtime.market_data_stale_poll_seconds=60
runtime/runner.py tracks last_entry_bar_update_time in state diagnostics
any new closed entry bar from any deployed symbol resets the stale clock
if no closed entry bars arrive for 180 minutes, runner prints:
  MARKET_DATA_STALE no closed entry bars for <minutes> minutes; possible holiday/weekend/feed issue
runner also logs MARKET_DATA_STALE to events_{YYYYMM}.csv
warning repeats no more often than runtime.heartbeat_minutes while stale
when market_data=STALE, runner slows the outer poll sleep from 5 seconds to 60 seconds
when a closed entry bar appears again, runner prints/logs MARKET_DATA_RESUMED and returns to 5-second polling
ALIVE heartbeat includes market_data=OK/UNKNOWN/STALE
```

Clock clarification:

```text
runtime.poll_interval_seconds=5 controls how often the bot checks MT5 for new closed bars
runtime.heartbeat_minutes=15 controls only how often the terminal prints ALIVE
the bot will not miss 5m bars because signal detection is tied to the 5-second poll, not the 15-minute heartbeat
```
