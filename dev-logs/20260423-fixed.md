# 2026-04-23 V2 Runtime Fix Notes

## Broker UTC Offset / False Stale Market Fix

Problem observed on VPS:

```text
SYMBOL_DATA_ERROR ... no closed bars in probe
MARKET_DATA_STALE no closed entry bars for 600+ minutes
```

The one-shot market probe showed MT5 was returning bars successfully, but some
bar timestamps appeared about three hours ahead of Python UTC:

```text
now_utc              = 2026-04-23T03:44:24Z
AUDUSD 1min close    = 2026-04-23T06:45:00Z
status               = FUTURE_OR_FORMING_ONLY
mt5_last_error        = (1, 'Success')
```

Root cause:

V2 treated MT5 bar timestamps as true UTC and filtered closed bars with local
Python UTC. The broker server clock can be UTC+2 or UTC+3 depending on DST, so
fresh MT5 bars were rejected as future bars.

V1 had a broker offset diagnostic in `lecagy/run_data.py`, but V2 had not
carried the broker-server-time concept into the new cache.

## Daily Telegram Status

Legacy V1 sent a daily London-open Telegram status message via
`notifier.london_open(state)`. V2 initially migrated only trade-open
notifications, so quiet days had no Telegram proof that the bot was alive.

V2 now supports:

```yaml
notifications:
  enabled: true
  paper_trades: false
  live_trades: true
  daily_status: true
  daily_status_utc_hour: 7
```

Behavior:

```text
send once per UTC day at or after 07:00 UTC
persist diagnostics.last_daily_status_date to avoid duplicate sends
skip in dry-run
respect notifications.enabled and notifications.daily_status
```

Message contents:

```text
mode
deployment
symbols
equity / balance
open trades / portfolio cap
broker position count
market-data status
latest entry bar
GPS status
```

The V2 daily status intentionally excludes Highwind, Rule2, and CB.

Runtime behavior now:

```text
detect broker UTC offset from symbol_info_tick("EURUSD")
print/log BROKER_TIME_OFFSET on startup
pull closed bars from MT5 start_pos=1
normalize broker-server bar timestamps back to UTC
seed MARKET_DATA_RESUMED during warmup if current entry bars exist
normalize broker history deal close_time with the same offset
```

The important startup line:

```text
Broker UTC offset: UTC+3 (OK)
```

The offset may become UTC+2 when broker DST changes. It is detected on each
startup.

## Why `start_pos=1`

V2 no longer decides whether a bar is closed by comparing MT5 timestamps to the
local Python clock.

MT5 bar positions are used instead:

```text
start_pos=0  current/forming bar
start_pos=1  latest closed bar
```

This keeps the no-future-leak rule intact and avoids false future-bar rejection
when broker server time is not true UTC.

## Reconciliation Close Time Fix

After the cache fix, broker-close reconciliation also needed the offset.

Without this, `history_deals_get(position=ticket)` close times could be logged in
broker-server time and GPS could see shifted or future `close_time` rows.

Fix:

```text
BrokerReconciler.set_broker_utc_offset(offset_hours)
_deal_time(..., broker_utc_offset_hours=offset_hours)
```

Trade rows now keep:

```text
open_time   Python UTC time when the bot records the open
close_time  normalized UTC time from broker deal history
bar_time    normalized UTC closed entry bar time
```

## VPS Log Reset

The old VPS logs were generated before the broker UTC offset fix. They can
mislead `status.py`, GPS, and manual diagnosis because they contain stale-market
noise and pre-fix probe rows.

Recommended VPS cleanup:

```powershell
cd C:\SystemC
Remove-Item -Path .\logs\portfolio_option_a_202604\*.csv -Force
Remove-Item -Path .\logs\portfolio_option_a_202604\gps\*.csv -Force
Remove-Item -Path .\logs\portfolio_option_a_202604\gps\*.md -Force
```

Keep:

```text
C:\SystemC\state_live_portfolio_v2.json
C:\SystemC\.ennv
C:\SystemC\config\
```

There is no persistent V2 bar cache to clear:

```text
data.cache.persist_to_disk: false
```

Restarting the process rebuilds the in-memory bar cache from MT5.
